# ai-job-rating 要件定義書

求人情報をAIで抽出・スコアリングし、自分の希望条件に照らしてランキングするアプリケーションの要件定義。

- ステータス: ドラフト（v0.1）
- 最終更新: 2026-06-20
- 関連ドキュメント: [開発ロードマップ](./roadmap.md)

---

## 1. 背景・目的

求職活動では多数の求人を比較する必要があるが、各求人が自分の希望（年収・働き方・技術スタックなど）にどれだけ合致するかを横断的に・定量的に判断するのは手間がかかる。

本アプリは、求人ページのURLを投入するとAIが内容を構造化抽出し、ユーザーが設定した**重み付き条件**に照らして自動でスコアリング・ランキングする。これにより求人比較と優先順位付けを効率化する。

### 設計の原則

- **個人セルフホスト前提のOSS**: 本人が使うだけでなく、リポジトリをフォークした第三者が自分の環境へ簡単にデプロイし、自分の希望条件で利用できることを要件とする。
- **抽出とスコアリングの分離**: AIによる構造化抽出（高コスト・求人1件につき原則1回）と、重み付きスコアリング（軽量・決定的・重み変更のたびに再計算）を分離する。これにより重みを変えても**AIを再実行せず即座に再ランキング**できる。

---

## 2. 想定ユーザーとユースケース

- **想定ユーザー**: 求職活動中の本人（技術職を主に想定）。1人＝1デプロイのシングルユーザー運用。
- **代表ユースケース**:
  1. フォークして自分のCloudflareアカウントにデプロイする。
  2. 自分の希望条件（評価項目の希望値と重み）をUIで設定する。
  3. 気になった求人のURL（詳細ページ／一覧ページ）を投入する。
  4. AIが内容を抽出・保存し、スコアを算出する。
  5. ダッシュボードでスコア順に求人を比較し、項目別の内訳を確認する。
  6. 重みを調整して再ランキングし、自分の優先度に合った並びを得る。

---

## 3. スコープ

### スコープ内

- 求人URLの手動投入（単一詳細URL／一覧URL／HTML貼り付け）
- ページ取得（SSR／SPA対応、段階的に認証下ページ対応）
- AIによる構造化抽出
- 重み付きスコアリングと再計算
- スコア順ランキング表示と項目別内訳
- 重み・希望条件の設定UI
- 企業評判の取得とスコアへの加味（Phase 2、[§7.2](#72-企業評判reputationの取得方針phase-2)）

### スコープ外（現時点）

- マルチユーザー／アカウント管理・認可
- 自動巡回クローリング（定期的な全サイト巡回）
- 求人サイト公式API連携
- 応募管理・進捗トラッキング・通知

> スコープ外項目のうち将来的に検討するものは [開発ロードマップ](./roadmap.md) の Phase 2 に記載する。

## 4. 機能要件

| ID | 機能 | 概要 | フェーズ |
| --- | --- | --- | --- |
| F-1 | 求人投入 | 単一詳細URL／一覧URL／HTML貼り付け（フォールバック）で求人を投入する | PoC（単一URL・HTML貼り付け）／ MVP（一覧URL） |
| F-2 | ページ取得 | SSRは `fetch`、SPA/認証下はBrowser Renderingで取得。一覧/詳細を判定し、一覧なら詳細URLを抽出 | PoC（SSR）／ MVP（SPA・一覧抽出・認証） |
| F-3 | AI抽出 | 取得HTMLを構造化JSON（求人スキーマ）へ変換 | PoC |
| F-4 | スコアリング | 構造化フィールド × ユーザー設定（希望値＋重み＋任意のハードフィルタ）で項目別スコアを算出し加重合計。AI非依存で再計算可能 | PoC（固定設定）／ MVP（UI設定） |
| F-5 | 永続化 | 投入求人・抽出結果・スコア・設定をDBに保存 | MVP |
| F-6 | ランキング表示 | スコア順の一覧と項目別内訳を表示 | MVP |
| F-7 | 重み設定UI | 項目ごとの重み・希望値を編集し、即時に再ランキング | MVP |
| F-8 | 企業評判 | 企業単位で口コミスコア・件数を取得（**Claude API web search を本命**、URL/HTML投入・手入力で補助）し、独立カテゴリとしてスコアに加味。対象サイトは設定で永続化 | Phase 2 |

### F-1 求人投入の詳細

- **単一詳細URL**: 1件の求人詳細ページ。
- **一覧URL**: 検索結果や求人一覧ページ。詳細ページへのリンクを抽出し、各詳細を個別に処理する（F-2）。
- **HTML貼り付け**: 取得が困難なケース（取得失敗・SPA・認証）の**フォールバック**。PoCでは抽出品質の検証手段としても使う。

### F-2 ページ取得の詳細

- **SSR（サーバーレンダリング）ページ**: `fetch` でHTMLを取得。
- **SPA（クライアントレンダリング）ページ**: Browser Rendering（ヘッドレスブラウザ）でレンダリング後のDOMを取得。
- **一覧/詳細の判定とリンク抽出**: 取得ページが一覧か詳細かを判定し、一覧なら詳細ページURL群を抽出する（ヒューリスティック＋必要に応じてAI）。
- **認証下ページ（MVP）**: ユーザーがCookie/セッションをサーバーに渡し、Browser Renderingがその状態で取得する。資格情報の取り扱いは非機能要件（§8）とリスク（§10）で扱う。

## 5. スコアリング設計

### 5.1 評価項目（初期セット）

ユーザーが項目ごとに「希望値」と「重み」を設定する。下記は初期セット（フォーク先で増減可能）。実在の求人ページの構造化ラベルに基づき、報酬だけでなく**働き方（残業・休日）**まで評価できるよう設計する。

**報酬 (compensation)**

| 項目 | 例・補足 | 正規化タイプ |
| --- | --- | --- |
| 想定年収 | 希望レンジ（例: 700万〜）。レンジ求人は下限/上限/中央値で評価 | 数値レンジ |
| 月給・基本給 | 年収未記載時の補助。固定残業代を含むか要注意 | 数値レンジ（補助） |
| 賞与 | 有無・回数（例: 年2回） | カテゴリ／有無 |
| 昇給 | 有無・頻度（例: 年1回） | カテゴリ／有無 |
| 退職金 | 有無 | 有無 |

**働き方・ワークライフバランス (work-life balance)**

| 項目 | 例・補足 | 正規化タイプ |
| --- | --- | --- |
| 残業 | 時間外労働有無・固定（みなし）残業時間（例: 30時間）。多いほど減点 | 数値／カテゴリ |
| 年間休日 | 例: 122日。多いほど加点 | 数値レンジ |
| 休日制度 | 完全週休2日制（土日祝）等 | カテゴリ一致 |
| 有給取得率 | 記載があれば加味（例: 74.7%） | 数値レンジ |
| リモート可否 | フルリモート／一部／出社 | カテゴリ一致 |
| フレックス・裁量労働 | 有無 | 有無 |

**勤務条件 (working conditions)**

| 項目 | 例・補足 | 正規化タイプ |
| --- | --- | --- |
| 勤務地 | 希望エリアとの一致 | カテゴリ一致 |
| 雇用形態 | 正社員／契約／業務委託 | カテゴリ一致 |
| 雇用期間 | 無期／有期 | カテゴリ一致 |

**仕事内容・スキル (role & skills)**

| 項目 | 例・補足 | 正規化タイプ |
| --- | --- | --- |
| 技術スタック一致 | 希望スキル集合との一致度 | 集合一致／AI判定 |
| 必須要件の充足 | 募集の必須要件を自分が満たすか | AI判定 |
| 歓迎要件の充足 | 歓迎要件をどれだけ満たすか | AI判定 |
| 業界・事業ドメイン | 希望ドメインとの適合 | カテゴリ／AI判定 |
| 言語要件 | 日本語のみ／英語必須 | カテゴリ一致 |

**企業属性 (company attributes)**

| 項目 | 例・補足 | 正規化タイプ |
| --- | --- | --- |
| 企業規模 | 従業員数（単体/連結） | 数値レンジ |
| 企業フェーズ | 上場区分（上場/未上場/上場準備） | カテゴリ |

**企業評判 (company reputation)** — Phase 2 で追加（詳細は [§7.2](#72-企業評判reputationの取得方針phase-2)）

| 項目 | 例・補足 | 正規化タイプ |
| --- | --- | --- |
| 口コミ総合評価 | OpenWork等の5点満点スコアを0〜100へ換算 | 数値レンジ |
| 口コミ件数 | 信頼度の重み（少件数はサブスコアを減衰/低信頼フラグ） | 信頼度係数 |
| サブ項目（任意） | 待遇/風通し/成長/WLB など | 数値レンジ |

> 企業評判は**求人単位ではなく企業単位**の属性。求人の所属企業に紐付けてキャッシュし、同一企業の全求人で再利用する。スコアへは独立カテゴリ「企業評判」として加重合算する。

> 上記はあくまで初期セット。実際の求人ページにはこれ以外にも多数の項目（試用期間・面接回数・受動喫煙防止措置・服装など）が含まれるため、抽出スキーマはこれらも**取れるだけ取って保存**しておき、ユーザーが後から評価項目に追加できるようにする（[§5.3](#53-抽出とスコアリングの分離再掲重要) の分離方針と整合）。

### 5.2 スコア算出方式

- 各項目について、AI抽出値とユーザーの希望値から **0〜100 のサブスコア** を算出する。
- 正規化は3類型で扱う:
  - **数値レンジ**: 希望値に対する達成度を連続値で算出（例: 希望年収を超えれば100、下回るほど逓減）。
  - **カテゴリ一致**: 希望カテゴリとの一致／部分一致で離散的に算出。
  - **AI判定**: 抽出値だけで機械的に判定しにくい項目（要件充足・ドメイン適合など）を、抽出フェーズで0〜100相当の判定値として得る。
- 総合スコア = Σ(サブスコア × 重み) ／ Σ(重み)（正規化加重平均）。
- **ハードフィルタ（任意）**: 必須条件（満たさなければ除外）・除外条件（該当すれば除外）を項目に設定でき、ランキングから除外またはフラグ表示する。
- **情報なし（unknown）の扱い**: 実際の求人ページは未記載項目を `-` 等で返す。値が取れない項目は、加点も減点もせず**加重合計の分母から外す**（中立）。UI上は「情報なし」と明示し、ハードフィルタ対象なら除外せずフラグ表示にとどめる。
- **ラベルの正規化**: 同じ意味の項目でも求人サイトごとにラベルが異なる（「想定年収/年収」「時間外労働/みなし残業」等）。抽出フェーズでサイト依存のラベルを**正規スキーマのキーへ寄せて**保存し、スコアリングは正規キーのみを参照する。

#### 5.2.1 構造的に誤りやすい categorical キーの定義（#88）

spike #65 / PR #72 の live 実測で、モデル非依存に揃って誤る categorical キーが判明した（companyPhase / workLocation / techStack / holidaySystem / businessDomain）。原因は「何を正解とするか」がコード/プロンプト側で未定義だったこと。各キーの意味・型・正規化方針を以下に確定する（実装の単一ソース）。

| キー | 意味（抽出対象） | 正規化方針 | canonical 許容値 |
| --- | --- | --- | --- |
| `companyPhase` | 企業の**上場区分**（設立年は対象外） | コード側で決定的に canonical 化 | `listed`（上場・東証各市場）/ `preIpo`（上場準備・IPO準備）/ `private`（未上場・非上場） |
| `holidaySystem` | 休日**制度**の区分（年間休日数は `annualHolidays` の責務で対象外） | コード側で決定的に canonical 化 | `fullTwoDayWeekoff`（完全週休2日）/ `twoDayWeekoff`（週休2日）/ `shift`（シフト・交代制）/ `fourWeekEightOff`（4週8休） |
| `workLocation` | 勤務地の地名 | 開集合のため canonical 化せず生表記を保持 | （なし。プロンプト/description で「地名を簡潔に」と明示） |
| `techStack` | 使用技術の列挙 | 開集合のため canonical 化せず生表記を保持 | （なし。プロンプト/description で「原文の技術名を列挙のまま・要約/翻訳しない」と明示） |
| `businessDomain` | 事業ドメイン・業界 | 開集合のため canonical 化せず生表記を保持 | （なし。プロンプト/description で「事業ドメイン/業界を簡潔に」と明示） |

- canonical 化するキー（companyPhase / holidaySystem）は `extract.ts` の `CATEGORY_RULES` に決定的規則を持たせ、LLM 依存を減らす（§7.1 / #65 決定方針）。マッピングに無い値は生表記を 1 カテゴリとして残し情報を捨てない。
- canonical トークン名は既存規約（`full` / `partial` / `onsite` 等）に合わせ英語小文字キャメル。

### 5.3 抽出とスコアリングの分離（再掲・重要）

- 抽出フェーズの出力（構造化JSON）と、AI判定が必要な項目の判定値は**保存しておく**。
- スコアリングは保存済みの抽出結果とユーザー設定のみから決定的に計算する。
- よって**重み・希望値の変更ではAIを再実行しない**。再取得・再抽出が必要になるのは、ページ内容が変わった場合や評価項目の定義自体を変えた場合のみ。

## 6. データモデル（概念）

実装時のスキーマ詳細はPhase 1で確定する。概念モデルは下記。

- **jobs**: `id`, `source_url`, `source_type`(detail/listing/paste), `status`(fetched/extracted/scored/failed), `fetched_at`, `company_id`, 生HTMLへの参照
- **extractions**: `job_id`, 構造化JSON（求人スキーマ）, `model`, `extracted_at`
- **criteria_config**: `criterion`, `desired_value`, `weight`, `hard_filter`(none/required/exclude)（ユーザー設定、シングルユーザー前提）
- **scores**: `job_id`, `criterion`, `sub_score`, および総合スコア（キャッシュ）
- **companies**（Phase 2）: `id`, `name`, `houjin_bangou`(法人番号・任意), 名寄せキー
- **reputation_snapshots**（Phase 2）: `company_id`, `source`, `overall_score`, `review_count`, サブ項目JSON, `fetched_at`（企業単位でキャッシュ）
- **reputation_sources**（Phase 2）: 対象口コミサイトの設定（`name`, `base_url`/識別子, 取得方式）。設定画面で永続化

ストレージ割り当ての想定:

- **D1**: jobs / extractions / scores / criteria_config などの構造化データ
- **R2**: 生HTML・スクリーンショットなどの大きめのオブジェクト
- **KV**: 軽量な設定・キャッシュ（必要に応じて）

## 7. AIモデル方針

### 7.1 求人抽出（Workers AI）

#### 候補モデル（Cloudflare Workers AI、2026-06時点で確認）

| モデル | Model ID | コンテキスト | 関数呼び出し | 備考 | 価格（入力/出力 per Mトークン） |
| --- | --- | --- | --- | --- | --- |
| gpt-oss-120b (OpenAI) | `@cf/openai/gpt-oss-120b` | 128,000 | 対応 | reasoning対応 | $0.35 / $0.75 |
| llama-4-scout-17b-16e-instruct (Meta) | `@cf/meta/llama-4-scout-17b-16e-instruct` | 131,000 | 対応 | **Vision対応**・Batch対応 | $0.27 / $0.85 |

> 上表は初期の候補。**評価対象モデルの最新セットと live 実測の手順は [#65 spike](./spikes/issue-65-extraction-mechanism.md)** に集約し、性能評価（精度・速度・コスト・504）で既定モデルを確定する。

#### 用途と方針

- **構造化抽出（F-3）**: HTML本文を入力し、求人スキーマのJSONを得る。**構造化の機構は JSON Mode に固定せず差し替え可能なアダプタ**（JSON Mode / function calling / prompt+検証+修復）とし、出力形式は機構非依存のコードレイヤ（スキーマ検証 → 修復 → 正規化）で担保する。**使用モデルと既定機構は性能評価で決定する**（[#65 spike](./spikes/issue-65-extraction-mechanism.md)）。
- **一覧/詳細の補助判定・リンク抽出（F-2）**: ヒューリスティックで判定しきれない場合にAIを併用する。
- **日本語品質の検証**: 日本語求人に対する抽出精度・速度・コストを候補モデル横並びで live 実測し、既定モデル・既定機構を選定する（評価対象と手順は [#65 spike](./spikes/issue-65-extraction-mechanism.md)）。
- **Vision活用の可能性**: llama-4-scout はVision対応のため、SPA/認証下でDOM取得が難しいページのスクリーンショット抽出にフォールバックできる可能性がある（将来検討）。
- **コスト最小化**: AI抽出は求人1件につき原則1回。結果を保存して再利用し、重み変更では再実行しない（§5.3）。長いHTMLは本文抽出・トリミングで入力トークンを削減する。

### 7.2 企業評判（reputation）の取得方針（Phase 2）

求人を出している企業の評判を定量化してスコアに加える機能。Phase 2 で追加する。

**データソースの現状（2026-06 時点で一次確認）**

| ソース | 定量データ | 公開API | 取得制約 |
| --- | --- | --- | --- |
| OpenWork | 総合評価(5点)+8項目+残業/有給 | なし | ログイン依存 |
| エン ライトハウス / 転職会議 | 口コミ評点(5点系) | なし | ログイン依存・ToS |
| Google Maps（Places API） | rating(5点)+件数 | あり（公式・従量） | 店舗評価で従業員評価ではない |
| 国税庁 法人番号 Web-API | 名寄せ用（スコアではない） | あり（公式・無料/要ID） | 企業名↔法人番号のマッチング基盤 |

**取得方式**

総合スコアや件数の手入力は現実的でないため、評判の検索・収集は **Claude API を主軸** とする。

- **Claude API による評判検索**。Claude API の **`web_search` サーバーツール**（`web_search_20260209`、動的フィルタリング内蔵）で企業名から口コミスコアをWeb検索し、**構造化出力**（`output_config.format`）で総合スコア・件数・サブ項目のJSONを得る。ユーザー操作は不要で、求人から取得した**企業名（可能なら法人番号）をキー**に自動取得する。
- デフォルトモデルは `claude-opus-4-8`（コスト次第で `claude-sonnet-4-6` / `claude-haiku-4-5` も候補）。
- **補助/フォールバック**: 評判ページのURL/HTMLを投入してAIが数値抽出する経路を残す（検索が外す場合や特定サイトを指定したい場合）。スコア・件数の手入力は**任意の上書き手段**にとどめ、主たる入力経路にはしない。
- 対象とする口コミサイト・優先順位は**設定画面で永続化**する（[§6](#6-データモデル概念) の `reputation_sources`）。認証下サイトをサーバー取得する場合は、求人取得と同じ Cookie/セッション＋Browser Rendering を流用できる（ToSリスクは利用者責任、§10）。
- Claude API キー（`ANTHROPIC_API_KEY`）は評判機能の前提。未設定時は評判カテゴリを中立扱いで除外し、URL/HTML投入・手入力で補完する。

**名寄せ・スコア統合**

- 企業名は表記揺れがあるため、可能なら**法人番号 Web-API で正規化**して企業を一意化し、評判スナップショットを企業単位でキャッシュする。
- スコアは独立カテゴリ「企業評判」として加重合算（§5.1）。**件数による信頼度**を考慮し、少件数の高評価が支配しないよう減衰または低信頼フラグを付す。データなしは中立（§5.2）。

## 8. 非機能要件

- **フォーク容易性**: `wrangler.jsonc`（バインディング定義）、`.dev.vars.example`（必要シークレットの雛形）、READMEのセットアップ手順、最小限の依存で「フォーク→設定→デプロイ」が短時間で完了すること。
- **コスト管理**: AI呼び出し回数・トークン量を抑える設計（抽出1回・キャッシュ・本文トリミング）。Cloudflareの無料枠/低コスト枠での個人運用を想定。
- **取得マナー**: robots.txt の尊重、取得間隔・レート制御、適切なUser-Agent。一覧展開時の同時取得数に上限を設ける。
- **エラーハンドリング**: 取得失敗・抽出失敗時はHTML貼り付けフォールバックへ誘導し、`status` に失敗を記録する。
- **資格情報の取り扱い**: 認証下対応で受け取るCookie/セッションは最小権限・最小保持。保存方針はリスク（§10）で扱う。
- **秘匿情報の非コミット**: APIキー（`ANTHROPIC_API_KEY` 等）・Cookie/セッション・トークンなどの秘匿情報はリポジトリにコミットしない。`.gitignore` と `.dev.vars.example`（値は空の雛形）で誤コミットを防ぎ、実値は Wrangler secrets／環境変数で投入する。
- **アカウント固有設定の外部化**: 秘匿情報でなくとも、個人のアカウント固有の値（CloudflareアカウントID、D1/R2/KVのID・バインディング名など）はコードに直書きせず、環境変数・`wrangler.jsonc`・設定ファイルで投入する。フォーク先が自分の値に差し替えるだけで動く状態を保つ。
- **再現性**: 同一入力・同一設定なら同一スコアになる（スコアリングは決定的）。
- **開発手法（TDD）**: t-wada メソッドの TDD（Red → Green → Refactor）で開発し、スコアリングなど決定的ロジックはユニットテストで担保する。

## 9. アーキテクチャ概要

スタックは **Cloudflare Workers 単体（Hono + 静的資産）**。フォークとデプロイが最も簡単な構成を採用する。

```
[ユーザー] --URL投入--> [Worker (Hono)]
                          |
        +-----------------+------------------+
        | SSR: fetch        SPA/認証: Browser Rendering
        +-----------------+------------------+
                          v
              [一覧/詳細 判定・リンク抽出]
                 |（一覧なら詳細URLへ展開）
                 v
              [AI抽出 (Workers AI / 機構アダプタ + 検証修復)]
                 v
        [保存: D1(構造化) / R2(生HTML)]
                 v
        [スコアリング (決定的・重み付き)]
                 v
        [ランキングUI / 重み設定UI (Hono)]
```

- 複数詳細ページの非同期処理にはCloudflare Queuesの利用をMVPで検討する。
- 構成要素（取得層／抽出層／スコアリング層／保存層／UI）は責務を分離し、それぞれ単体でテスト可能にする。

| リスク | 内容 | 対応 |
| --- | --- | --- |

## 11. UI・デザイン方針（MVP以降）

- **デザインシステムの管理・可視化**: MVP以降、[Claude Design](https://claude.ai/design) でデザインシステム（カラー・タイポグラフィ・コンポーネント等）を管理・可視化し、UIの一貫性を保つ。

- **UIテストとスクリーンショット**: UI実装時は Playwright 等のブラウザテストと、PR貼り付け用のスクリーンショット撮影を検討し、UIの回帰を検知する（§8 TDDと整合）。

## 12. 用語集

- **抽出 (extraction)**: 求人ページのHTMLから、構造化されたフィールド（年収・勤務地など）をAIで取り出すこと。
- **スコアリング (scoring)**: 抽出結果とユーザー設定（希望値・重み）から、項目別スコアと総合スコアを決定的に計算すること。
- **ハードフィルタ (hard filter)**: 満たさない／該当する場合にランキングから除外する必須・除外条件。
- **SSR / SPA**: サーバーレンダリング型 / クライアントレンダリング型のページ。後者はヘッドレスブラウザでのレンダリングが必要。
- **Browser Rendering**: Cloudflareのヘッドレスブラウザ機能。SPAや認証下ページの取得に用いる。
- **JSON Mode**: Workers AIの構造化出力機能。出力をJSONスキーマに沿わせる。本アプリでは固定機構ではなく、差し替え可能な構造化抽出アダプタの一つ（[§7.1](#71-求人抽出workers-ai)・[#65 spike](./spikes/issue-65-extraction-mechanism.md)）。
- **Claude Design**: デザインシステムを管理・可視化するツール（`claude.ai/design`）。MVP以降、UIのデザイントークン・コンポーネント管理に用いる。
