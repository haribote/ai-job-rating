# ai-job-rating 開発ロードマップ

要件定義（[requirements.md](./requirements.md)）を3フェーズに分けて段階的に実装する計画。各フェーズは独立して価値を持ち、完了の定義（DoD）を満たしたら次へ進む。

- 最終更新: 2026-06-22

## Phase 0 — PoC（技術検証）

**目的**: Cloudflare Workers AI の**日本語求人に対する抽出品質**を検証し、デフォルトモデルを選定する。最小経路でスコアリングまで通す。

### スコープ

- 公開ページ（認証不要）の**単一詳細URL**を対象（一覧展開はしない）。
- 取得は SSR（`fetch`）のみ。HTML貼り付けをフォールバック兼検証手段として用意。
- Workers AI（`@cf/openai/gpt-oss-120b` / `@cf/meta/llama-4-scout-17b-16e-instruct`）で構造化抽出（JSON Mode）。
- 評価項目・希望値・重みは**コード/JSON固定**（UIなし）。
- 結果は単一求人のスコアと項目別内訳をその場で表示（永続化なし、または最小限）。

### 主要タスク

- [x] Workers + Hono の最小プロジェクト雛形（`wrangler.jsonc`、Workers AI バインディング）
- [x] `fetch` によるHTML取得と本文抽出・トリミング
- [x] 求人スキーマ（JSON）の定義
- [x] JSON Mode を使った構造化抽出の実装
- [x] 固定設定での重み付きスコアリング（決定的計算）
- [x] 単一結果の表示（スコア＋項目別内訳）
- [x] HTML貼り付けフォールバック
- [x] 両モデルの日本語抽出精度を複数求人で比較・記録

### DoD

- 公開求人URLを入力すると、抽出→スコア→内訳表示まで通る。
- 両候補モデルの抽出品質を比較し、デフォルトモデルを決定・記録した。
- 求人スキーマと正規化方式の妥当性を実データで確認した。

### 振り返り（実施: 2026-06-22）

**成果**: 主要タスク 8 件すべて完了、マイルストーン 15/15 issue クローズ。DoD（公開URL→抽出→スコア→項目別内訳表示の貫通）を達成。決定的スコアリング＋unknown 中立（§5.2）と、抽出↔スコアリングの責務分離（§5.3）を実装で確立した（重み変更で AI 再実行が不要なことを確認）。

**モデル選定（根本原因と決定）**: roadmap 候補の `gpt-oss-120b`（reasoning モデルで JSON Mode 非準拠・`content=null`）と `llama-4-scout`（JSON Mode 公式保証外でスキーマ網羅が弱い）が**両方とも JSON Mode 抽出に不適合**と判明。実装(#11)で唯一の公式 JSON Mode 対応 `@cf/meta/llama-3.3-70b-instruct-fp8-fast` を採用した判断は妥当と確認し、暫定デフォルトとして継続。弱点は遅さ・504 タイムアウト・context 24k。詳細・一次ソースは [docs/spikes/issue-15-model-comparison.md](./spikes/issue-15-model-comparison.md)。複数求人での精度裏取りは、実サンプルが PII を含み実質1社しか集まらず未完（再定義/保留）。

**正規化統合 gap（#59）**: 決定的スコアリング自体は完成したが、抽出の正規化がナイーブだと実データでサブスコアが縮退する問題が判明。年収の単位（円↔万円）と categorical の canonical 化（生 JP → `full`/`partial` 等）は #64 で解消。aiJudged（スキルマッチ）は判定基準未確定のため Phase 0 では unknown 中立扱いとし暫定方針を記録（`src/extract.ts`）。

**実データ品質の教訓**: trimHtml の制御文字/生山括弧/名前付き実体（#55）、parseNumbers の注記ノイズ・transient 504 リトライ（#57）の堅牢化を通じ、「SSR 取得でも実データは想定以上に汚い」ことを確認。取得・トリミング層の防御が抽出品質に直結する。

**Phase 1 への仕分け**:

- 反映/起票済み: **#65** — JSON Mode に依存せず日本語求人を「早く・安く」構造化抽出する手段の検討（spike）。JSON Mode 固定はモデルを公式対応リストに縛り速度・コスト・context・504 の制約を受けるため、データモデル確定前に方針を固める。
- 繰り越し（起票済み）: **#68** — aiJudged スキルマッチの実値化（#59 gap3）。希望スキル集合との突合方式を決定。関連 #65（抽出機構）/ #19・#20（希望条件・重みの設定UI）。
- 見送り/保留: 複数求人での網羅的なモデル精度比較は、抽出機構の方針（#65）が確定するまで保留（当初の比較設計自体を再定義）。

## Phase 1 — MVP（実用最小版）

**目的**: 永続化・ランキング・重み調整UI・一覧/SPA/認証対応を備えた、日常的に使える求職ツールにする。

### スコープ

- 永続化（D1 / R2）と投入求人のランキング表示。
- 重み・希望条件の設定UI（変更で即再ランキング、AI再実行なし）。
- **一覧URL**からの詳細URL抽出と各詳細の処理。
- **SPA対応**（Browser Rendering）。
- **認証下ページ対応**（Cookie/セッションをサーバーに渡す方式）。
- 複数詳細ページの非同期処理（Cloudflare Queues を検討）。

### 主要タスク

- [x] D1 スキーマ確定（jobs / extractions / scores / criteria_config）と R2 連携（生HTML）
- [x] 投入求人のランキング一覧UI（スコア順・項目別内訳）
- [x] 重み・希望条件の設定UI と再スコアリング（決定的・AI非依存）
- [x] 一覧/詳細の判定・詳細URL抽出（ヒューリスティック＋必要に応じAI）
- [x] Browser Rendering による SPA 取得
- [x] 認証下取得（Cookie/セッション受け渡し、最小権限・最小保持）
- [x] 複数詳細の非同期処理（Queues 検討）とレート/同時実行制御
- [x] 取得・抽出失敗時のエラーハンドリングとフォールバック導線
- [x] デザインシステムを Claude Design で管理・可視化し、トークン/スタイルをリポジトリ（CSS等）へ反映（実行時依存にしない）
- [x] Playwright 等による UI テストとスクリーンショット撮影（PR 貼り付け用）の検討・導入
- [x] フォーク用整備（`.dev.vars.example`、README セットアップ手順、デプロイ手順）

### DoD

- 詳細URL・一覧URL・認証下ページのいずれからも求人を取り込める。
- 求人がスコア順に一覧表示され、項目別内訳が確認できる。
- 重みを変更すると AI を再実行せずに即座に再ランキングされる。
- フォークした第三者が README 手順に従って自分の環境にデプロイ・利用できる。

### 振り返り（実施: 2026-06-25）

**成果**: sub-issue 17 件 ＋ トラッキング #70 すべてクローズ、DoD 4 項目達成。着手 2026-06-20 〜 #70 close 06-25 の約 6 日。D1/R2 永続化・スコア順ランキング・設定UI（重み変更で即再ランキング・AI非再実行）・一覧/SPA/認証取得・フォーク用整備を live 検証済み。src 約 10.5k LOC / テスト 26 ファイルで、決定的スコアリング・unknown 中立（§5.2）・抽出↔スコアリング分離（§5.3）・フォーク容易性を実装で維持した。

**wave-rider 並列オーケストレーションの効果と限界**: 依存トポロジに沿って 06-22 の約 2 時間で Wave 2–4（#71/#76/#77/#79/#80/#81/#82/#83）を集中 merge し、独立タスクの並列化が明確に効いた。一方で限界も確認: (1) subagent の自己申告が不正確（biome 見落とし・テスト数誤り・node_modules staging）で orchestrator による offline checks 再実行が必須、(2) 実 Cookie・モデル真値判定など人手リソース依存のタスクは並列化の外に残る。

**抽出品質は「後追いハードニング」になった（根本原因）**: 実データが想定以上に汚く・多様で、抽出ロジックが実データ投入後に繰り返し堅牢化を要した。Phase 0 の #62/#63（注記ノイズ・transient 504・制御文字・named entity）に続き、Phase 1 でも #88/#90 で**モデル非依存の構造的抽出エラー**（companyPhase/holidaySystem を全候補が同一キーで揃って誤る）を DoD 後に検出・是正した。決定的 canonical 化で companyPhase/holidaySystem は是正したが、businessDomain 等の開集合キーは canonical 化せずプロンプト依存の非ブロック残課題として残置。教訓: 抽出品質は golden サンプルでの構造的検証を spike/早期に組み込まないと後追いになる。

**「オフライン素通り・live でのみ露見」クラスのバグ**: wrangler の変数 import 未バンドル（runtime "No such module"）、positional entry の compat date 既定（binary 超過）など、unit test / dry-run を抜けて live deploy でのみ露見するバグが複数発生した。教訓: deploy 後の最小 live スモーク（binding 解決・dynamic import）を検証導線に組み込む。

**人間ゲートの滞留は #72 に集中**: live 検証を要した draft のうち #75（認証 fetch）/#78（Queues）は実セッション検証後 06-23 に merge（滞留 1 日）。一方 #72（抽出機構 spike・モデル選定）はモデル精度の真値判定が人手依存で 06-22→06-25 と 3 日滞留し、最終クローズの長尾になった。教訓: 主観判定・実資格情報を要するタスクは早期にスケジュールし、DoD 充足後も明示キューで追跡する。

**Phase 2 への仕分け**:

- 反映/起票候補: **live スモークの仕組み化**（runtime-only バグの検出を deploy 導線に組み込む）、**抽出品質ゲートの前倒し**（golden サンプルでの構造的検証を spike 段階から）。
- 繰り越し（残課題）: businessDomain 等の**開集合キーの正規化方針**（#88 の非ブロック残課題）。
- 既知の罠の自動化: lockfile の tab インデント → 2-space 正規化を pre-commit hook 化（#74 でも再発）。

## Phase 2 — 企業評判 ＋ 将来拡張

MVP 運用後に着手。優先度は必要性に応じて判断する。

### 企業評判機能（F-8、要件定義 §7.2）

**目的**: 求人を出している企業の評判を定量化し、独立カテゴリ「企業評判」としてスコアに加える。

#### 主要タスク

- [ ] **Claude API による評判検索** — `web_search` サーバーツール＋構造化出力で、企業名（可能なら法人番号）から総合スコア・件数・サブ項目を自動取得（デフォルト `claude-opus-4-8`）
- [ ] `ANTHROPIC_API_KEY` の設定導線（評判機能の前提）と、取得元・スコアのUI明示
- [ ] 企業の名寄せ（companies）— 表記揺れの正規化、可能なら国税庁 法人番号 Web-API で一意化
- [ ] 評判データモデル（reputation_snapshots / reputation_sources）と企業単位キャッシュ
- [ ] 対象口コミサイト・優先順位を設定画面で永続化
- [ ] 補助/フォールバック経路 — 評判ページURL/HTML投入→AI抽出、任意のスコア手入力上書き
- [ ] 件数による信頼度の重み付けと、独立カテゴリとしてのスコア加重合算
- [ ] データなし時の中立扱い・低信頼フラグのUI表示

#### DoD

- Claude API により、企業名から口コミ総合評価・件数を自動取得でき、企業単位でキャッシュされる。
- 取得元とスコアがUIに明示され、誤取得時はURL/HTML投入・手入力で補正できる。
- 評判が独立カテゴリとして重み付き合算され、件数の少ない高評価が支配しない。
- `ANTHROPIC_API_KEY` 未設定時は評判カテゴリを中立で除外し、他項目のスコアリングは成立する。

### その他の将来拡張（検討項目）

- 求人サイト公式API連携（取得の安定化・規約遵守）
- 定期再取得（求人の鮮度更新・クローズ検出）
- ブラウザ拡張／ブックマークレットによる取得（認証下の取得を簡便化）
- Vision モデルによるスクリーンショット抽出フォールバック
- マルチユーザー化（アカウント・認可）
- 応募管理・進捗トラッキング・通知

## フェーズ間の依存

- Phase 0 で確定した**求人スキーマ**と**正規化方式**が Phase 1 のデータモデル・スコアリングの土台になる。
- Phase 1 の取得層（SSR/SPA/認証）の抽象化が Phase 2 の取得手段拡張の土台になる。
